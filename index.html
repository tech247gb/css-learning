The 5-Point UI Diagnostic Checklist — Use This on Any Screen
1
Hierarchy check — can you read the screen in 3 seconds?
Cover the screen, uncover it, and look for 3 seconds. What do you read first? If you can't tell, there's no hierarchy. The heading should be obvious, the action should be obvious.
2
Spacing check — squint and look for rhythm
Blur your eyes. Does the whitespace feel consistent? Elements that belong together should be closer together. Related groups need more space between them than within them.
3
Color check — does every color have a job?
Point to each color and ask: what does this communicate? If the answer is "nothing" or "decoration", remove it. Red = error/danger only. Green = success only. Blue = primary actions only.
4
Consistency check — are same elements styled the same?
Find all buttons, all inputs, all cards, all headings. Do they match? One inconsistency breaks the whole design. Inconsistency signals "someone wasn't paying attention."
5
Contrast check — is all text readable?
Gray text on white background is the #1 accessibility failure. Check: body text on background (4.5:1 minimum), placeholder text, disabled states, small text. Use the WebAIM contrast checker.

Prompting AI for Design — The Art of Getting Great Output
From generic to production-quality: how you write the prompt changes everything
60 min ▼
The 5 Elements of a High-Quality UI Prompt
1
Context — what is this, who uses it?
"A login form for a B2B finance SaaS" produces better output than "a login form". AI needs to know the product category, user type, and tone.
2
Framework — be exact, not vague
"React + Tailwind CSS" not "React". "HTML + vanilla CSS (no frameworks)" not "HTML". Exact framework = exact output format.
3
Visual style — give it a reference point
"Clean, minimal, light theme, no gradients, flat design" or "Reference: Linear.app's UI style". AI has seen millions of UIs — give it a direction.
4
Required states — list them explicitly
Default, hover, focus, loading, error, disabled, success. AI will skip states you don't mention. Always list the ones you need.
5
Constraints — what NOT to do
"No external libraries", "no gradients", "no animations", "no shadows". Negative constraints are as important as positive ones — they prevent AI from defaulting to generic patterns.
Prompt Templates — Copy, Adapt, Use
🧱 New Component
Use when building any individual UI component from scratch.
Build a [component] using [React/HTML] + [Tailwind/CSS]. Context: [app type], users are [who]. Style: [clean/minimal/dark/colorful], [light/dark] theme. Include states: default, hover, focus, loading, error. Constraints: no external libs, accessible, mobile-first. Color: primary #[hex], border-radius: [value].
🖥 Full Page Layout
Use when designing an entire screen or view.
Design a [page name] for a [app type] app. Users: [who, what they're trying to do]. Key sections: [list sections — header, sidebar, main, etc]. Framework: [React/HTML + Tailwind/Bootstrap]. Layout: [grid/sidebar/full-width/split-panel]. Tone: [professional/minimal/friendly]. Mobile-first. No lorem ipsum — use realistic data.

Do's and Don'ts — Prompting for UI
✓ Do This
  • Specify exact framework and CSS approach
  • Describe who the users are and the app context
  • List all states you need explicitly
  • Build one component at a time, then iterate
  • Reference real apps: "Like Linear.app style"
  • Ask AI to explain design choices — learn from it
  • Always run the 5-point checklist on AI output
✗ Avoid This
  • Vague style: "make it look modern/nice/clean"
  • Asking for an entire app in one prompt
  • Accepting first output without reviewing it
  • Skipping accessibility requirements
  • Using output without understanding what it does
  • Asking AI "what's wrong?" without sharing the code
  • Overriding an AI fix without asking why it was done

4 Session
Designing a Full App with AI — The Complete Workflow
A repeatable 6-step process from blank page to production-quality UI, solo
60 min ▼
The 6-Step Design-Independent Workflow
1
Define — screens, users, core action
Before any code or prompt, write down: what app, who uses it, what is the one thing they do most. AI amplifies your thinking — it can't replace it.
I'm building a [app type] for [users]. Their primary goal is [main action]. Help me identify the 3–5 screens this app needs and what each screen must accomplish. Output as a simple list.
2
Design system — tokens before components
Get consistent spacing, colors, and typography defined first. This is the one prompt that prevents all future inconsistency.
Create a CSS design token system for a [app type]. Define: primary/secondary/neutral color palette, 6-level type scale (px values), spacing scale (4px base), border-radius values, shadow definitions. Style: [minimal/professional/friendly]. Output as CSS custom properties (:root { --color-... }).
3
Layout shell — structure before content
Build the skeleton of each page with empty placeholder containers before adding any real components or content.
Create the layout shell for the [page name] page. Layout: [top nav + sidebar + main content / other]. Framework: [React/HTML + Tailwind/CSS custom properties from step 2]. No real content yet — just named placeholder divs. Labels: .layout-header, .layout-sidebar, .layout-main, etc. Must be responsive: single column on mobile.
4
Components — one at a time, always
Never ask AI to build an entire page in one prompt. One component, one prompt, then review before moving to the next. This is how you stay in control.
Build a [component] using the design tokens from step 2. States: default, hover, focus, loading, error, disabled. Props: [list]. Accessibility: visible focus ring, aria-labels, semantic HTML. Use the CSS variables I defined (--color-primary, etc). Output only the component — no page wrapper.
5
Review — apply the 5-point checklist every time
After each component, run your eyes through hierarchy, spacing, color, consistency, contrast. Fix before moving on. Don't build on top of broken foundations.
Review the component above using these 5 checks: 1. Hierarchy: is reading order clear in 3 seconds? 2. Spacing: are gaps consistent with a 4px scale? 3. Color: does every color have a semantic purpose? 4. Consistency: does it match the other components? 5. Contrast: are all text/background combos at 4.5:1+? List issues with exact fixes. Be specific.
6
Audit — consistency pass across the whole app
Once all screens are assembled, do one final scan for consistency. This is where good design systems pay off — if step 2 was done well, this pass is quick.
Audit these [N] screens for consistency: [paste or describe each screen briefly] Check for: - Mixed button styles or padding - Inconsistent heading sizes between pages - Different card border-radius values - Spacing that doesn't follow the 4px scale - Any new colors not in the original design tokens Prioritized list of fixes, most impactful first.
When AI Output Is Bad — Fix It, Don't Restart
Output is generic → add reference apps and more style context to the prompt Inconsistent spacing → paste your design tokens and say "use only these values" Breaks on mobile → add the responsive add-on template from Session 3 Accessibility missing → append the accessibility add-on template Code is too long / complex → ask for one piece at a time Never regenerate from scratch if targeted iteration can fix it — you lose context
